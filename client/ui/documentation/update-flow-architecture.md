# Архитектура процесса обновления (Update flow)

## Текущий поток (как было)

1. **Проверка обновления**: `checkUpdates(version)` → сервер → `onUpdateCheckResult(POSSIBLE_UPDATE | REQUIRED_UPDATE | ...)`.
2. **Старт обновления** (клик по кнопке или REQUIRED_UPDATE):
   - Останавливается core, убираются панели.
   - Вызывается **`startUpdate(osType)`** в потоке UI:
     - **`getFilePathsWithHashes()`** — рекурсивный обход директорий и **хэширование каждого файла** (включая большие). Это блокирует поток UI на десятки секунд.
     - Формируется JSON и отправляется пакет `UPDATE_ACCEPT`.
   - **После возврата** `startUpdate` показывается диалог обновления.
3. Сервер присылает `UPDATE_METADATA` (список файлов на скачивание/удаление).
4. Скачивание файлов → `onLoadingProgress(0..100)` → `onUpdateLoaded(emptyUpdate)`.
5. Применение: logout при необходимости, запуск update applier.

### Проблемы текущей архитектуры

- **Блокировка UI**: расчёт хэшей выполняется в том же потоке, что и UI → окно «висит» до завершения.
- **Нет фазы «подготовка»**: этап сканирования/хэширования не отделён от «отправка манифеста» и «скачивание», нельзя показать прогресс подготовки.
- **Один синхронный метод**: `startUpdate` делает и тяжёлую работу (хэши), и быструю (отправка). Нет возможности отменить подготовку (например, по кнопке «Выход»).
- **Слабое разграничение ошибок**: сбой при хэшировании и сетевая ошибка обрабатываются одинаково (`onNetworkError`).

## Предлагаемая архитектура

### Фазы обновления (явный контракт)

| Фаза | Кто выполняет | Где выполняется | Обратная связь в UI |
|------|----------------|-----------------|----------------------|
| **Preparing manifest** | updater::Client | Фоновый поток | «Подготовка… N/M файлов» (+ опционально имя текущего файла) |
| **Sending manifest** | updater::Client | Тот же фоновый поток (коротко) | Без смены статуса или «Отправка списка…» |
| **Downloading** | NetworkController | Сетевой поток | `onLoadingProgress` → «Загрузка… X%» |
| **Applying** | UpdateManager | UI | «Перезапуск…» |

### Разделение ответственности

- **updater::Client**:
  - **Подготовка манифеста** (тяжёлая работа): сбор списка файлов и расчёт хэшей в фоновом потоке с вызовом **прогресса** (текущий/всего, путь файла).
  - **Отправка манифеста**: построение JSON и отправка пакета (быстро).
  - Не знает о Qt; прогресс пробрасывается через `EventListener::onManifestProgress` (вызов может приходить из фонового потока — UI сам маршалит в главный поток).

- **EventListener** (мост updater → UI):
  - Добавлен **`onManifestProgress(current, total, currentPath)`** — может вызываться из фонового потока.
  - Остальные колбэки без изменений.

- **UpdateManager** (логика UI):
  - Сначала **показывает диалог** обновления и ставит статус «Подготовка…».
  - Вызывает **`startUpdateAsync(osType)`** (не блокирует UI).
  - В слоте **`onManifestProgress`** обновляет текст/прогресс («Подготовка… N/M файлов»), маршалируя вызов в главный поток через `QMetaObject::invokeMethod`.
  - При появлении **`onLoadingProgress`** переключает статус на «Загрузка…» и показывает процент.

### Асинхронный запуск обновления

- **`startUpdateAsync(OperationSystemType)`**:
  - Возвращает управление сразу.
  - В отдельном потоке:
    1. Собирает список путей для сканирования (без хэшей).
    2. Для каждого файла: вызывает `onManifestProgress(current, total, path)`, затем считает хэш.
    3. Строит JSON и отправляет `UPDATE_ACCEPT`.
  - Дальше всё как раньше: ответ сервера → скачивание → `onLoadingProgress` / `onUpdateLoaded`.

### Опциональное расширение (отмена)

- В `Client` можно хранить `std::atomic<bool> m_manifestCancelled`.
- В цикле хэширования проверять флаг и при установке выходить без отправки пакета.
- В UI по кнопке «Выход» во время фазы «Подготовка» вызывать `cancelManifestPreparation()` и закрывать диалог.

## Итог изменений по слоям

1. **client/updater**
   - `getFilePathsWithHashes(optional<ManifestProgressCallback>)`: сначала сбор списка путей, затем цикл с вызовом прогресса и хэшированием.
   - `startUpdateAsync(OperationSystemType)`: поток + вызов `getFilePathsWithHashes` с прогрессом → отправка пакета.
   - Сохранение существующего синхронного `startUpdate` не обязательно; при желании можно оставить для тестов.

2. **EventListener + UpdaterEventListener**
   - Новый метод `onManifestProgress(size_t current, size_t total, const std::string& currentPath)`.

3. **UpdateManager**
   - Показ диалога и установка статуса «Подготовка…» **до** вызова `startUpdateAsync`.
   - Новый слот `onManifestProgress` → обновление текста/прогресса в диалоге.
   - При первом `onLoadingProgress(> 0)` — смена статуса на «Загрузка…».

В результате UI не блокируется на этапе подсчёта хэшей, пользователь сразу видит диалог и прогресс подготовки, затем — прогресс загрузки.
