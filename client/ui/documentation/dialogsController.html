DialogsController Logic and Behavior

UI dialogs with internal logic and user interaction (not notifications).

Overview

DialogsController manages real dialogs that contain user-facing interaction logic and/or complex UI: confirmation flows, selection dialogs, settings dialogs, and call-related popups.

DialogsController should not manage notification banners; those belong to NotificationController.

Responsibility Split

DialogsController

  • Creates and positions dialogs that contain logic and controls.
  • Owns (or coordinates) overlays for modal-like dialogs.
  • Connects dialog signals to application logic (e.g., close requests, selections).

NotificationController

  • Shows lightweight status banners and transient messages.
  • Handles stacking and auto-hide behavior.

Dialogs Managed

The following remain in DialogsController because they contain user interaction logic:

  • UpdatingDialog (update progress/status UI)
  • ScreenShareDialog (screen selection, cancellation)
  • AudioSettingsDialog (device selection, sliders, mute state)
  • FirstLaunchDialog (welcome experience)
  • AlreadyRunningDialog (multi-instance warning)
  • IncomingCallDialog (accept/decline flows)

Execution Sequence (typical dialog)

  1. Caller requests a dialog via a show* method.
  2. Controller ensures the overlay exists and is visible (if required).
  3. Dialog is created (if not already) and parented to overlay or main window.
  4. Dialog is positioned and raised.
  5. Signals from the dialog are connected to controller or external handlers.
  6. On hide, dialog is disconnected, hidden, and scheduled for deletion.

Signals

DialogsController exposes signals to bubble user intent and dialog events upward:

  • closeRequested()
  • screenSelected(int), screenShareDialogCancelled()
  • incomingCallAccepted(QString), incomingCallDeclined(QString), incomingCallsDialogClosed(QList<QString>)
  • Audio settings related signals (device selection, volumes, mute, refresh)

Important Rules

  • Do not put notification-like UI into DialogsController.
  • Dialogs with timers/auto-hide should generally be moved to NotificationController if they are status-only.
  • Dialogs that contain buttons, choices, or application flow must remain here.
