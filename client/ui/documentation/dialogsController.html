DialogsController behavior and rules

UI notification stacking rules and pending operation behavior.

Overview

This document describes how DialogsController decides which notification dialogs are visible, what state is stored internally, and how pending operations are stacked.

Notification categories

Managed notifications stacked state

  • ConnectionDownWithUser
  • PendingOperation

Non-stacked notifications

  • ConnectionDown (global connectivity down)
  • ConnectionRestored (global connectivity restored)
  • ConnectionRestoredWithUser (call participant restored)

Other dialogs

Updating, ScreenShare, AlreadyRunning, FirstLaunch, AudioSettings, IncomingCalls, etc. are not affected by the rules in this document.

Core rules

Only one notification dialog is visible at a time.
When a new notification is shown, the current one is hidden.
For managed notifications, only state is stored (type, key, text), not dialog instances.
When a managed notification is hidden and older managed notifications exist, the most recent stored one is shown again.

Managed notification details

ConnectionDownWithUser

  • Added to the stack (last wins).
  • If already visible, text is updated and it is raised.
  • On hide: removed from the stack, hidden if visible, then previous managed notification is restored.

PendingOperation

  • Added to the stack with an optional operation key.
  • The key allows multiple pending operations to be tracked independently.
  • If the visible pending operation has the same key, its text is updated.
  • Hiding with a key removes only that entry; if it was visible, it hides and restores the previous state.

Non-stacked notification details

ConnectionDown

  • Clears the managed stack.
  • Hides any current notification and shows ConnectionDown.
  • When hidden, the last managed notification is restored if any.

ConnectionRestored and ConnectionRestoredWithUser

  • Not added to the stack.
  • When shown, they hide the current notification.
  • When hidden, the last managed notification is restored if any.

Pending operation keys and usage

  • Managers create a separate timer per operation key (core::UserOperationType).
  • When the timer fires, they call showPendingOperationDialog(text, key).
  • When an operation completes or fails, they call hidePendingOperationDialog(key).
  • This prevents one operation from canceling another operation's pending state.
